# ADR 001: Use Kotlin Multiplatform for Shared Bitcoin Logic

## Status

Accepted

## Context

Apricot needs a layer that handles Bitcoin domain models, API integration, data mapping, caching, and use cases. This logic does not depend on any platform UI and could, in principle, run on multiple targets (iOS, Android, server-side).

The iOS app is built with SwiftUI. Writing the entire backend-of-frontend in Swift would work but would duplicate effort if an Android version were ever added. The alternative is to share the logic in a cross-platform module from the start.

## Decision

Use Kotlin Multiplatform (KMP) to implement the shared module. The shared module owns:

- Domain models (address summary, transactions, inputs/outputs)
- DTOs and mappers (Mempool.space response → domain)
- Repository interfaces and implementations
- Use cases
- In-memory cache
- API client (Ktor)

The iOS app consumes the shared module as an XCFramework, generated by Gradle. XcodeGen references the framework from a local path.

## Consequences

**Positive:**
- Business logic is written once and testable in isolation via `./gradlew :shared:allTests`, without needing a simulator.
- Domain models, mappers, and use case behavior are covered by fast JVM tests.
- If an Android target is added, the shared module can be reused without rewriting the core.

**Negative:**
- The build chain is more complex: Gradle must run before Xcode can compile. A first-time contributor must run `make bootstrap` and understand why.
- KMP interop with Swift has rough edges. Value classes are unboxed in Swift and lose their type identity. Coroutine-based APIs require an iOS-facing facade to bridge the async boundary cleanly — the facade wraps KMP calls and exposes a Swift-native `async`/`throws` interface to the rest of the app.
- Debugging across the Kotlin/Swift boundary is harder than debugging pure Swift.
- Kotlin and Xcode toolchains can conflict on version updates, requiring careful coordination.

## Alternatives Considered

**Swift-only implementation:** Simpler toolchain, easier Xcode integration, but all logic would be Swift-only. Any future Android port would require a full rewrite of domain logic.

**Swift Package + shared Swift logic:** Could work for iOS/macOS, but offers no path to Android. Not relevant for this project's current scope, but a factor in the decision.
